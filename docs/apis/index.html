Internal · Integration reference

APIs we can integrate, sorted by whether they actually work here.

The only question that matters for a site with no backend: does the API send Access-Control-Allow-Origin so a browser can call it directly? Everything below was tested with that lens on 18 July 2026, not copied from documentation.

Two honesty flags run through this page. First: "government" means the data is officially published — but some useful APIs are third-party wrappers around government data, not government-run. We mark those. Second: CORS status changes and demo keys get rate-limited, so re-test before building. Our own scripts/qa.py --links won't catch an API that quietly changes its CORS policy.

✅ Usable from a static page today

CORS-open, no per-user login. Callable straight from the visitor's browser.

Only data.gov.in is government-run. The others are convenient wrappers — fine to use, but say so, and don't imply official endorsement. For PIN and IFSC there are no official CORS-open government endpoints, which is why wrappers exist.

⛔ Real, official, but need a backend

These return a specific citizen's private data, so they require a server, a registered secret, and (mostly) MeitY/NeSL approval. They cannot run from GitHub Pages — and shouldn't, because a secret in browser code is not a secret.

API Setu is 2,791 consent-based "Docs" vs 109 "Services" — almost none callable from a static page. This is the arpit@commercios.com + MeitY route already on the roadmap. It needs a minimal backend (even a free Cloudflare Worker) before any of it is reachable.

The rule of thumb

Green light

Public data, CORS-open

Build it client-side. Prices, PIN, IFSC, maps. Show the source and the date.

Yellow

Public data, no CORS

Needs a tiny proxy (free Cloudflare Worker) to add CORS headers. Cheap, but it's a backend.

Red

Anyone's private data

Backend + registered secret + approval, always. Never attempt from the browser.
